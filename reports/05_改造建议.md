## 三、改造建议（按优先级）

### P0 — 统一 LLM 调用入口（必须改）

**问题**：Mother 有 4 条 LLM 调用路径，内置 TokenPool 副本绕过服务。

**方案**：
```
Mother 中新建 app/llm/provider.py:

  class ProviderManager:
      def chat(messages):
          1. 尝试 TokenPool HTTP API (POST /v1/chat/completions)
          2. TokenPool 不可用时 fallback 到本地环境变量
          3. 返回 OpenAI 格式响应

  MotherRuntime._build_candidates() → 删除
  MotherRuntime.run() → 改用 ProviderManager.chat()
  LLMClient → 改用 ProviderManager
  app/token_pool.py → 删除
```

**影响范围**：
- `mother_runtime.py` 的 `_build_candidates()` 方法
- `llm.py` 的 `__init__` fallback 逻辑
- `llm_router.py` → 删除

### P1 — 统一工具注册（重要）

**问题**：三套工具系统，定义分散在 4 处。

**方案**：
```
新建 app/tools/registry.py:

  class ToolRegistry:
      def register(name, handler, category, description)
      def dispatch(name, arg) → result
      def list_all() → [ToolDef]

  merge:
      tools.py BUILTIN_TOOLS → registry
      skills.py GitHub/SSH → registry
      LLM_SKILL_PROMPTS → registry (作为 prompt-only 技能)
```

**影响范围**：
- `tools.py` → 重构为 ToolHandler 模式
- `skills.py` → 注册到 registry
- `mother_runtime.py` 的 `_execute_tool()` → 改用 registry.dispatch()
- `tool_runtime.py` → 删除

### P2 — 控制面板数据统一（改进）

**问题**：10+ JSON 文件 + SQLite 并行，数据源分散。

**方案**：
```
将 admin JSON 数据迁移到 SQLite:

  admin.json          → admin_config 表
  admin_sessions.json → admin_sessions 表
  users.json          → api_users 表 (或合并到 sessions)
  stats.json          → 使用 TokenPool call_log 聚合
  keys.json           → 使用 TokenPool keys 表
  bugs.json           → bugs 表
  features.json       → features 表
  version.json        → version_config 表

保留 JSON 的:
  heartbeat_logs/     (高频写入，SQLite 不适合)
  miclaw_instances.json (MiClaw Bridge 状态)
```

### P3 — 安全加固（增强）

| 项 | 当前 | 建议 |
|----|------|------|
| Bridge 鉴权 | 无 | 增加 X-Bridge-Key header 验证 |
| 设备命令签名 | 无 | 增加 HMAC 签名 |
| CORS | `*` | 限制到已知 origin |
| Cookie | SameSite=Lax | SameSite=Strict |
| 上传 Token | 硬编码 | 支持多 Token + 轮换 |
| 审计日志 | 无 | 管理操作写入 audit_log 表 |

### P4 — 架构增强（未来）

| 项 | 说明 |
|----|------|
| EventBus | 进程内事件总线，Mother ↔ Panel 解耦 |
| Session 持久化 | WorkingMemory 序列化到 DB，支持热重启 |
| 结构化日志 | JSON 格式日志，接入可观测平台 |
| API 版本化 | `/v1/`, `/v2/` 前缀 |
| 前端模块化 | panel.js 拆分，或迁移到轻量框架 |

## 五、改造路线图

```
Phase 0 ✅  Architecture Freeze (当前)
    │
Phase 1     删除废弃代码 (~39 文件)
    │       创建 ProviderManager 统一 LLM 入口
    │       Mother 通过 HTTP 调用 TokenPool
    │
Phase 2     统一工具注册 (ToolRegistry)
    │       合并 tools.py + skills.py + tool_runtime.py
    │
Phase 3     控制面板数据迁移 (JSON → SQLite)
    │       安全加固 (Bridge 鉴权, CORS, 审计)
    │
Phase 4     架构增强 (EventBus, Session 持久化, 日志)
    │
Phase 5     商业化: TokenPool 计费+支付集成
```

| 阶段 | 工作量 | 风险 |
|------|--------|------|
| Phase 1 | 3-5 天 | 低（删代码 + 加 HTTP 客户端） |
| Phase 2 | 3-5 天 | 低（重构工具 dispatch） |
| Phase 3 | 5-7 天 | 中（数据迁移需兼容旧格式） |
| Phase 4 | 5-10 天 | 中（新组件引入） |
| Phase 5 | 10-20 天 | 高（支付集成） |
